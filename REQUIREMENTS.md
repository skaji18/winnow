# Winnow — 要件・設計メモ

> AI実行つきタスク管理ツール。ただし本質は「タスクを片付ける道具」ではなく
> **あなたの判断アテンションを配給する道具**である。

---

## 0. 一行コンセプト

> 登録されたタスクの木を、AIが「自動でやる／あなたに上げる／人間がやる」に確信度つきで仕分け、
> 自動分は実行まで回し、**あなたのキューにはエスカレーションだけが残る**。
> そして道具は、あなたの普段のさばきを教師信号にして、その仕分けの境界を日々較正していく。

この一行に、以下すべてが畳まれている。

---

## 1. 設計の背骨（これを外すと別物になる）

1. **希少資源は実行能力ではなく、人間の判断アテンションである。**
   1人＋多数エージェントの世界では実行はいくらでも増える。律速はあなたの注意。
   よって製品の最深の仕事は「処理量を増やす」ことではなく「あなたの注意の落とし所を最適化する」こと。

2. **委譲するのは「決定」ではなく「アテンション配分」。**
   AIに「人間/AI」の二択を断定させない（＝確信のない所で断定させると、一番効くノードから静かに人間が抜ける）。
   代わりに **確信度 × ステークス × 不可逆性** でスコアリングさせ、低リスク・可逆・高確信だけ自動処理し、残差を上げる。
   人間の仕事は「全件を仕分ける」から「**境界線を監督する**」に縮む。

3. **状態はエージェントの記憶ではなく、システム・オブ・レコード（タスク管理ツール本体）に持たせる。**
   永続するのはバックログ・優先順位・依存・ステータス。これは決定論的に・無料で保持できる。
   エージェントは都度立ち上がる「無状態の認知」に徹する。

4. **学習は地味な簿記であって、全自動オプティマイザではない。**
   ゼロから方策を学ばない。明示ルールを先に置き、**残差だけ**を学ぶ。中身の大半はカテゴリ別の基準率補正（カウント）。

5. **UXで死ぬ。** ラベルが「いつものさばきの副産物」になっていなければ餅。
   "学習させるための操作"・別建てのダッシュボードを足した瞬間に回らなくなる。

## 2. ドメインモデル

### 2.1 ノードとリーフ（「タスク」を割る）

「タスク」と一括りにすると分類器も実行器も誤作動する。最低限この区別を持つ。

| 種別 | 正体 | 割れる? | 実行可能? | 例 |
|---|---|---|---|---|
| **ノード** | 問い／意図 | Yes | No（直接は不可） | 「この案件をどう進めるか」「方向性を決める」 |
| **リーフ** | 実行可能タスク | No | Yes | 「このエンドポイントを実装する」「このデータを移行する」 |

- 各アイテムは `kind: node | leaf` フラグを持つ（＝後述する抽象度ラダーの高度）。
- 木構造：ノードは子（ノード or リーフ）を持つ。リーフは葉。
- **一番ありがちな事故**：方向性未確定のノードをリーフと誤認して実行に流す。これを防ぐのが昇格判定（§3.3）。

### 2.2 抽象度ラダー

ノードは降りるにつれ抽象度が下がる：

```
霧を照らす（方向性が全く不明）
  → 戦略
    → 戦術
      → 具体手段
        → 実行（リーフ）
```

**重要な非対称：判断の重さと量はラダー上で逆向きに分布する。**

| | 上段 | 下段 |
|---|---|---|
| 量(ボリューム) | 極小 | 爆発する |
| 不可逆性 | 最大 | 小 |
| 判断の性質 | 還元不能に人間的 | 基準に照らせばチェック可能 |
| レビュー方式 | 人間が密に | 基準照合で薄く |

→ **人間の注意は上段へ寄せ、量は下段へ流す。** 上段への投資（鋭いスペック）は下段で複利で効く。
曖昧なスペックを下に渡すと、エージェントが方向性を再導出する羽目になり、そこが一番高くつき壊れる。

### 2.3 プロセス軸（ラダーと直交）

「どう降りるか」は別の自由度：**ウォーターフォール / イテレーティブ**。

- 原則：**サイクル長は、その段の不確実性に反比例させる。**
- 「何をどうやるか全く不明」な段でのウォーターフォールは有害（無知の上に計画をコミットする）。PoC/スパイクで**情報を買って**不確実性を潰す短サイクルが正解。
- 道が見えている実行段は、再計画不要なのでバッチ（ウォーターフォール的）に束ねる方が効率的。
- 合言葉：**「学ぶために回し、出すために束ねる」**。
- エージェントは反復が速くて安いので、人間だけの組織より最適点が「上段でもっと反復する」側に寄る（ただしクォータ天井あり）。

## 3. 機能要件

### 3.1 タスク管理（主・土台）

- アイテムの登録・編集・並べ替え・着手・却下・親子付け（＝既存のタスク管理ツールの基本動線）。
- 各アイテムが持つメタデータ（抜粋）：
  - `kind`（node/leaf）, `rung`（ラダー高度）, `parent`, `status`
  - `disposition`（auto / escalate / human）, `confidence`
  - `stakes`, `reversibility` スコア
  - `process`（waterfall/iterative）, `uncertainty_resolved`（bool）
  - 監査・履歴用：`auto_executed`, `human_overrode`, `audit_sampled`
- **新規の儀式はゼロ。** あなたが普段やる登録・並べ替え・着手・却下が、そのまま分類器の教師信号になる。

### 3.2 分類器（仕分け）

- **二値ではなく三値以上**：`auto / escalate / human`、各々に**確信度**を添える。
  「分からない」を表現できない分類器は、静かにあなたを盲点へ流す。
- スコアリング次元：**確信度 × ステークス × 不可逆性**。
- 仕分けの本当の切り口は「上段=人間」ではなく **レジビリティ（明示基準に乗るか）**：
  - AIが得意：基準ベースで照合できる次元、定型の自明案件を弾く、calibrated uncertainty の申告。
  - AIが構造的に盲：**unknown-unknown（霧そのもの）**、**文脈に宿るステークス**（顧客関係・戦略の二次効果など、AIが持たない文脈に依存する重さ）。
- 良い分類器＝レジブルな部分を安くさばき、**タシット(暗黙)な残差に人間を集中させる**装置。
- **アイテム本文は「指示」ではなく「観察対象データ」として扱う**（分類・分解・実行の各プロンプトで本文をデリミタで囲い、本文中の命令・スコア/disposition の自己申告には従わない）。本文が自分のスコア（確信度・ステークス・不可逆性・disposition）を自己指定していたら、それは過小エスカレーションを狙う**詐称シグナル**とみなし escalate に倒して確信度を下げる。背骨の「口はバカ、分類器が賢い」を機械的に強制する防御層。

### 3.3 木を割る系（3つに分割）

ユーザの素朴な「割る・割り方提案」機能は、実は3系統：

1. **分解器（Decomposer）**：ノードに効く。子（ノード/リーフ）を提案する。割り方の選択肢も出す。
2. **実行器（Executor）**：リーフに効く。実装・運用などの具体実行（§3.4）。
3. **昇格判定（Promotion Judge）**：出てきた子に「まだ問いか／もう実行可能か」を付け直す。
   これが無いと子が無限に問いのまま降りてこない。**割る→各子に実行可能?を付け直す、で一巡。**

### 3.4 実行とトリガー（"悩ましい"ところ）

- 自動実行は **可逆性で段を分ける**：
  - **可逆な実行 → 自動着火。**
  - **不可逆・高ステークス → 提案して人間ワンタップで承認。**
- **点火ゲート（方向性未確定の保護）**：可逆であっても、リーフが「方向性未確定のノードの配下」または「上流タスク未完」の状態なら自動着火せず **提案（承認待ち）** に倒す。具体的には、親ノードの不確実性が未解消／親が保留中／同一親の前順の兄弟（上流）が未完、のいずれかで発火する。これは §2.1 の「一番ありがちな事故（方向性未確定のノードを実行に流す）」を実行時にも止めるゲート。人間のワンタップ承認はこのゲートを通らない（人間が明示で押したものは流す）。
- **巻き戻し手順を実行ではなく提示する。** ソフトウェア実行ではワーカーに「変更ファイル一覧＋元に戻す具体手順（git の取り消しコマンド例）」を成果物として書かせる。道具はこれを自動実行せず、人間が取り消しを押したときに**手順として見せる**だけ（§4 の「安く取り消せる＆痕跡が残る」を具体化）。可逆と自己申告したのに巻き戻し手順が空なら、可逆性の過大評価とみなしそのカテゴリを締める。
- **承認＝外部送信のゴーサインの伝達（再拒否ループの解消）。** 不可逆・高ステークスを人間がワンタップ承認したら、ワーカーに「このアイテムに限り外部送信（push／PR作成）を実行してよい。ただしマージ・本番デプロイ・データ削除など**採用／破壊**にあたる最終操作はしない」を伝える。承認をゲート解除だけで終わらせず外部送信の許可まで運ぶ（承認しても push されない再拒否ループを断つ）。**PR作成＝可逆な提示／マージ＝不可逆な採用**の非対称を堅持し、採用は人間が外で行う。外部送信の解禁は既定 OFF・明示オプトイン（緩めは慎重 §3.6-3）。winnow 本体は git 操作を持たず実行主体はワーカー（その権限の技術的制約は別レイヤ）。
- **成果物の引き取り（handoff）。** 実行が成功しても成果物に人間の責任（レビュー／採用）が残るなら `done` に沈めず **「引き取り待ち」** で停止し、キュー前面に出す。人間が確認して**受け取る**と完了になる。引き取り要否は新しい申告軸を立てず、既存の **外部成果物の有無・可逆性・ステークス・外部送信の許可**から導出する（やって終わり＝ローカル日報等は従来どおり即完了）。気づきの口は能動通知を足さず**キューの前面化**で担保する（pull を尽くす。能動通知は §6／§7 の有料・常駐ゾーン）。これは §3.5 の「継ぎ目＝チェックポイント」を実行完了の局面に具体化したもの。
- 初期トリガーは **「人間がキューを開いた時に発火」** で十分実用。
- 常時稼働スケジューラは後から必要な所だけ（§6 の経済制約と直結：常駐＝有料ゾーン）。

### 3.5 レビューをパイプラインに戻す

- 実行後のレビューを**タスクとして登録**し、一連の処理に流す。
- 段と段の**継ぎ目（チェックポイント）が実装ポイント**：
  - 継ぎ目で学びを上段へ戻せば**イテレーティブ**、スペックを下へ渡すだけなら**ウォーターフォール**。
  - 同じ継ぎ目に、人間-in-the-loop ゲートと hooks/grader（自動レビュー層）が乗る。
  - **ゲート密度は上段で濃く、下段で薄く。**

### 3.6 自己較正（地味な簿記）

成立する版としない版がある。**「ログを溜める→MLが全自動最適化→ハンズオフ」は構造的に壊れる**（反事実が観測できない：自動に流した分の"もし見ていたら"は永遠に不明、自分が取った行動の分しかラベルが取れず、行動を選ぶ方策自身が見えるラベルを選ぶ）。成立条件は3つ：

1. **ゼロから方策を学ばない。** 明示ルール（このカテゴリは常にエスカレート/これは自動）を先に置き、残差だけ学ぶ。
   学習の中身の大半は**カテゴリ別の基準率補正**（例：「カテゴリXでAIのescalateを8割却下→Xは自動に倒す」というカウント）。浅くて頑健。
2. **監査サンプリングで盲点を割る。** 自動バケットの一定割合(N%)を、わざわざ人間レビューに回す。
   バイアスのない教師信号を生む唯一の手で、**節約したいアテンションを意図的に少額だけ払う**。これはタダにならない＝学習可能性の値段。
3. **信号の非対称を使う。**
   - 過剰エスカレーション（見たら自明＝AIで足りた）：**安く・速く・大量に**気づける。
   - 過小エスカレーション（自動に流したのが後で問題化）：**高く・遅れて・危険に**現れる。
   - → **締めるのは速く、緩めるのは慎重に。** デフォルト保守、ゆっくり開放、安全側に勝手に倒れる設計。
   - 「緩める（自動に倒す）」判定は点推定の却下率ではなく**小標本に頑健な下限**（観測割合の信頼区間下限）で行う。少数のサンプルで偶然高く出た却下率に釣られて早まって緩めないため。下限が閾値（8割）以上になって初めて倒す。母数は分類器の生提案（再調律（締め）で書き換える前の生提案）で数える——書き換え後を母数に混ぜると教師信号が汚染される。

## 4. UX要件（ここで生死が決まる）

**原則：ダッシュボードを作らない。あなたが既に開くキューを、仕分け済みにして返すだけ。**

利用動線：
新規アイテムがトラッカーに着く（既存の業務トリガー、新規儀式ゼロ）
→ 分類器が disposition・確信度・一行理由・スコアをメタデータで書く
→ あなたの「今日なに見る?」ビューが、火の海ではなく**エスカレーションだけの短いキュー**になる（自動分は畳む）
→ あなたが普段どおりさばく
→ **そのさばきが全部ラベルになる。**

必須UX要素：

1. **処分＝ラベル。** 各項目の操作は「やる／下段へ降ろす／分類し直す／"この種類はもう上げるな"」。
   後者2つが境界線への明示ナッジで、追加労力ゼロ。いつもの一手が信号になる。
2. **理由はグランス可能に。** 根拠を「読む」のに自分で判断するのと同じ時間がかかったらレバレッジ消滅。
   一行＋スコアバッジで「目に入る」長さ。**確信度は必ず出す**（出さないと全件裏取りされて終わる）。
3. **監査をキューに塩漬けで混ぜる。** 自動処理分のN%を、普段の項目と**見分けつかない形**で「30秒だけ確認」として差し込む。
   別画面の"監査タスク"にした瞬間に誰もやらない。
4. **自動は全部、安く取り消せる＆痕跡が残る。** fire-and-forget にしない。可逆＆可視。
   これが無いと怖くて境界を緩められず、レバレッジが出ない。
5. **ループを閉じて見せる。** 週次で一行：「今週：自動X / 上げY / あなたが覆したZ / カテゴリKを自動に倒した」。
   訂正が複利で効いているのが見えないと、人は監査税を払い続けない。

その他：
- **コールドスタート＝UXのJカーブ。** 初期は保守的＝全部エスカレートするのでキューは短くならない。
  「序盤は助ける感ゼロ、それは線を学んでいる音」と先に握る期待値管理が、計装より効くことがある。
- **再調律の取っ手。** エスカレーション率が膨らんだら締める／自動事故が出たら緩める、を一タップでできる粗いスライダー。
  最適点に着地する画面ではなく、動く標的を追う計器盤。

## 5. 非目標・既知の限界（正直に書く）

- **この最適化は収束しない。** 明示基準に乗る反復的な"真ん中"は速く食えるが、**新規・高ステークスの裾＝霧の領域**には原理的に届かない。蓄積が一番効かない場所と、一番重要な場所が一致する。
- **境界はドリフトする。** 案件・優先順位・「何が高ステークスか」が動くので、去年の境界は半分陳腐化する。保守コストはゼロにならない。
- **狙うバーは「正しさ」ではなく「現状(溺れてる)越え」**：
  「自分で全件目視するよりマシ、かつ誤仕分けからの回復が安い」で十分。最適境界を狙うと"言うは易し"に殺される。
- **最大の死因はJカーブ**：計装と監査税は、データも見返りも一番少ない初日に先払いする。複利が見える前に費用が来る。
- **較正データからの誤仕分け前提設計**：今のモデルの較正は完璧でないので、「仕分けが正しい前提」ではなく「誤仕分けからの安いリカバリ」を組み込む。

## 6. 経済・運用制約（サブスク前提との折り合い）

- **常設・スケジュール駆動の自律マネージャーをサブスクで安く回す道は、今のところ無い**（そこは構造的にAPI/Managed Agentsの有料ゾーン）。
  - 非対話的利用（自動ワークフロー等）は別クレジット枠＋API料金に分割されている。
  - Managed Agents はサブスク経由打ち切り済み、トークン＋セッション稼働時間で課金。
- よってサブスクの安さを活かす配置：
  1. **状態はトラッカー/DBに外部化**（無料・決定論的）。
  2. **manager/PjMの"思考"は、人間が座って回す対話的な儀式**にする（安いインタラクティブ枠を使う）。
  3. **実行＝対話的エージェント＋分野別サブエージェント、レビュー＝hooks/grader。**
  4. 夜間自動で優先順位づけ→ディスパッチさせたくなったら、**その層だけ**を有料ランタイムに切り出し、予算を別に持つ。
- **モデル階層化**：オーケストレータに上位、ワーカーに中位、整形に下位。並列ファンアウトはクォータを最速で溶かすので上限固定。

## 7. フェーズ計画（この道具自身を、この道具の原則で作る）

> 仕分けオプティマイザの構築自体が、§2-§3 のラダーに従う再帰になっている。

- **Phase 0 — 木と動線だけ**：ノード/リーフ区別つきのタスク管理＋手動さばき。学習なし。まず「いつものさばき」が溜まる器を作る。
- **Phase 1 — 最バカのルールベース分類器**：明示ルールのみで disposition を付与。確信度は固定。監査サンプル注入とラベル記録の動線を**最初から**入れる（後付け不可）。
- **Phase 2 — 基準率補正**：ログが「このカテゴリは安定」と証明した所だけ、カウントベースで境界を倒す。霧の部分は反復、固まった所はバッチ。
- **Phase 3 — 実行器＋可逆トリガー**：可逆リーフの自動着火、不可逆は提案＋ワンタップ。レビューのパイプライン還流。
- **Phase 4（必要な所だけ）— 常駐/スケジュール層**：有料ランタイムに切り出し、予算別管理。

## 8. 未決事項

- 監査サンプル率 N の初期値と、エスカレーション率に応じた動的調整則。
